# London School TDD for Persistent Feed System\n\nThis test suite implements **London School Test-Driven Development** for the agent feed persistent data system, focusing on mock-driven development and behavior verification.\n\n## 🎯 Methodology Overview\n\n### London School TDD Principles\n\n1. **Mock-First Approach**: Start by mocking all external dependencies\n2. **Outside-In Development**: Begin with acceptance tests and work inward\n3. **Behavior Verification**: Focus on interactions between objects, not state\n4. **Red-Green-Refactor**: Follow strict TDD cycles\n5. **Contract Testing**: Define clear interfaces through mock expectations\n\n### Key Differences from Classical TDD\n\n- **Mocks over Stubs**: Heavy use of interaction verification\n- **Design Focus**: Drives object collaboration design\n- **Isolation**: Each unit test is completely isolated\n- **Fast Feedback**: Tests run quickly due to extensive mocking\n\n## 📁 Test Structure\n\n```\ntests/tdd-persistent-feed/\n├── acceptance/          # Outside-in acceptance tests\n│   └── feed-experience.test.js\n├── unit/               # Mock-driven unit tests\n│   └── database-operations.test.js\n├── integration/        # API and service integration tests\n│   ├── api-endpoints.test.js\n│   └── frontend-integration.test.js\n├── performance/        # Load testing and performance verification\n│   └── load-testing.test.js\n├── regression/         # Comprehensive regression prevention\n│   └── regression-suite.test.js\n├── fixtures/           # Test data factories and builders\n│   └── test-data.js\n├── mocks/             # Sophisticated mock implementations\n│   └── database-mocks.js\n├── matchers/          # Custom Jest matchers for behavior verification\n│   └── behavior-matchers.js\n├── scripts/           # TDD workflow automation\n│   └── run-tdd-suite.js\n└── setup/             # Test environment configuration\n    └── jest.setup.js\n```\n\n## 🚀 Quick Start\n\n### Install Dependencies\n```bash\nnpm install\n```\n\n### Run Complete TDD Cycle\n```bash\n# Full Red-Green-Refactor cycle\nnpm run test:tdd\n\n# Individual phases\nnpm run test:tdd:red      # Write failing tests\nnpm run test:tdd:green    # Implement to pass\nnpm run test:tdd:refactor # Improve while maintaining tests\n```\n\n### Run Specific Test Suites\n```bash\n# Acceptance tests (outside-in)\nnpm run test:acceptance\n\n# Unit tests (mock-driven)\nnpm run test:unit:persistent\n\n# Integration tests\nnpm run test:integration:persistent\n\n# Performance tests\nnpm run test:performance:persistent\n\n# Regression suite\nnpm run test:regression:persistent\n```\n\n## 🔧 Test Configuration\n\n### Jest Configuration\n- **Mock Strategy**: Automock disabled, explicit mocking\n- **Test Environment**: Node.js for backend, jsdom for frontend\n- **Coverage Thresholds**: 85% statements, 85% branches, 90% functions\n- **Timeout**: 30 seconds for integration tests\n\n### Custom Matchers\n\nSpecialized matchers for behavior verification:\n\n```javascript\n// Verify interaction sequences\nexpect([mockA, mockB, mockC]).toHaveInteractionSequence([0, 1, 2]);\n\n// Verify database contracts\nexpected(query).toMatchDatabaseContract({\n  operation: 'SELECT',\n  parameters: ['param1', 'param2']\n});\n\n// Verify WebSocket message contracts\nexpect(message).toMatchWebSocketContract({\n  type: 'post:created',\n  data: { postId: 'test-id' }\n});\n\n// Verify performance thresholds\nexpect(metrics).toMeetPerformanceThresholds({\n  responseTime: 200,\n  memoryUsage: 512\n});\n```\n\n## 📋 Test Coverage Areas\n\n### 1. Database Operations\n- **Connection Management**: Pool lifecycle, error handling\n- **Query Execution**: Parameter binding, result processing\n- **Transactions**: ACID compliance, rollback scenarios\n- **Schema Integrity**: Table structure, constraints\n\n### 2. API Endpoints\n- **HTTP Methods**: GET, POST, PUT, DELETE operations\n- **Request/Response**: Payload validation, status codes\n- **Authentication**: Permission verification, security\n- **Error Handling**: Graceful degradation, proper error codes\n\n### 3. Frontend Integration\n- **Data Loading**: Initial fetch, loading states\n- **Real-time Updates**: WebSocket integration, live data\n- **User Interactions**: Create, like, comment, share\n- **Error Recovery**: Network failures, retry logic\n\n### 4. Performance\n- **Load Testing**: Concurrent users, high-volume data\n- **Memory Management**: Leak prevention, garbage collection\n- **Response Times**: API latency, UI responsiveness\n- **Scalability**: Resource utilization, bottleneck identification\n\n## 🎨 TDD Workflow\n\n### Phase 1: RED 🔴\n\n1. **Write Failing Test**\n   ```bash\n   npm run test:tdd:red\n   ```\n2. **Expected Outcome**: Tests fail (as designed)\n3. **Focus**: Define behavior contracts through mocks\n\n### Phase 2: GREEN 🟢\n\n1. **Minimal Implementation**\n   ```bash\n   npm run test:tdd:green\n   ```\n2. **Expected Outcome**: All tests pass\n3. **Focus**: Implement just enough code to satisfy contracts\n\n### Phase 3: REFACTOR 🔵\n\n1. **Code Improvement**\n   ```bash\n   npm run test:tdd:refactor\n   ```\n2. **Expected Outcome**: Tests continue to pass\n3. **Focus**: Optimize, clean up, improve design\n\n## 📊 Quality Gates\n\n### Coverage Requirements\n- **Statements**: ≥ 85%\n- **Branches**: ≥ 85%\n- **Functions**: ≥ 90%\n- **Lines**: ≥ 85%\n\n### Performance Requirements\n- **API Response Time**: ≤ 200ms (95th percentile)\n- **Database Query Time**: ≤ 50ms (average)\n- **Memory Usage**: ≤ 512MB (peak)\n- **Error Rate**: ≤ 1%\n\n### Reliability Requirements\n- **Test Pass Rate**: ≥ 95%\n- **Flakiness**: ≤ 5%\n- **Coverage Consistency**: No regression\n\n## 🔍 Mock Strategy\n\n### Database Mocks\n```javascript\nconst mockDatabase = {\n  connect: jest.fn(),\n  query: jest.fn(),\n  transaction: jest.fn()\n};\n\n// Verify behavior, not implementation\nexpect(mockDatabase.connect).toHaveBeenCalledTimes(1);\nexpect(mockDatabase.query).toHaveBeenCalledWith(\n  expect.stringContaining('SELECT'),\n  expect.arrayContaining(['param'])\n);\n```\n\n### Service Mocks\n```javascript\nconst mockFeedService = {\n  loadFeed: jest.fn(),\n  createPost: jest.fn(),\n  recordEngagement: jest.fn()\n};\n\n// Focus on collaborations\nexpect(mockFeedService.loadFeed).toHaveCollaboratedWith(\n  [mockDatabase, mockWebSocket],\n  expectedPattern\n);\n```\n\n## 📈 Continuous Integration\n\n### Pre-commit Hooks\n```bash\n# Run before each commit\nnpm run test:tdd\nnpm run lint\nnpm run typecheck\n```\n\n### CI/CD Pipeline\n1. **Lint & Type Check**\n2. **Unit Tests** (fast feedback)\n3. **Integration Tests**\n4. **Performance Tests**\n5. **Regression Suite**\n6. **Coverage Report**\n7. **Quality Gate Check**\n\n## 📝 Best Practices\n\n### Test Writing\n1. **Start with the behavior you want to verify**\n2. **Mock all external dependencies**\n3. **Use descriptive test names**\n4. **Keep tests focused and isolated**\n5. **Verify interactions, not implementation**\n\n### Mock Usage\n1. **Mock at the boundary of your unit**\n2. **Use spies for behavior verification**\n3. **Avoid over-mocking internal details**\n4. **Keep mock contracts simple and clear**\n5. **Update mocks when contracts change**\n\n### TDD Discipline\n1. **Never write production code without a failing test**\n2. **Write the simplest code that makes tests pass**\n3. **Refactor only when tests are green**\n4. **Commit after each successful cycle**\n5. **Keep cycles short (minutes, not hours)**\n\n## 🐛 Debugging\n\n### Common Issues\n\n1. **Mock Not Called**\n   ```javascript\n   // Check mock setup\n   expect(mockFn).toHaveBeenCalled();\n   \n   // Debug with mock calls\n   console.log(mockFn.mock.calls);\n   ```\n\n2. **Async Test Failures**\n   ```javascript\n   // Ensure proper async handling\n   await expect(asyncFunction()).resolves.toEqual(expected);\n   ```\n\n3. **Memory Leaks in Tests**\n   ```javascript\n   afterEach(() => {\n     jest.clearAllMocks();\n     jest.resetModules();\n   });\n   ```\n\n### Test Output Analysis\n\n```bash\n# Verbose test output\nnpx jest --verbose\n\n# Watch mode for development\nnpx jest --watch\n\n# Debug specific test\nnpx jest --testNamePattern=\"specific test name\"\n```\n\n## 📚 Resources\n\n### London School TDD\n- [Growing Object-Oriented Software, Guided by Tests](http://www.growing-object-oriented-software.com/)\n-
